Mikoto is a multi-service application. Running it in production means providing a set of backing infrastructure services (database, cache, object storage) alongside the application services themselves. This page covers what you need to configure, the exposed ports, essential environment variables, and recommended reverse proxy practices.

## Required services

Mikoto depends on the following infrastructure services. Each must be reachable by the application services before startup.

<CardGroup cols={2}>
  <Card title="PostgreSQL" icon="database">
    Primary relational database. Mikoto requires PostgreSQL 16 or later. Set `DATABASE_URL` to a standard `postgresql://` connection string.
  </Card>
  <Card title="Redis" icon="server">
    Used for pub/sub event delivery between service instances and for rate limiting. Set `REDIS_URL` to a `redis://` connection string.
  </Card>
  <Card title="MinIO / S3" icon="cloud">
    S3-compatible object storage for file uploads, attachments, and media. Any S3-compatible service (MinIO, AWS S3, Backblaze B2) works.
  </Card>
  <Card title="LiveKit (optional)" icon="video">
    Required only if you want voice and video channels. Omit `LIVEKIT_*` variables to disable voice/video features.
  </Card>
</CardGroup>

## Application service ports

All Mikoto application services use ports in the `351X` range. These are the ports your reverse proxy should route traffic to.

| Service | Default port | Description |
|---|---|---|
| Client | `3510` | Web client (React frontend) |
| Superego | `3511` | REST and WebSocket API server |
| Collab | `3512` | Real-time collaborative document editing |
| MediaServer | `3513` | Media upload and processing |

<Note>
  The Superego API (port 3511) serves both the REST API and the WebSocket endpoint at `/ws`. Route both HTTP and WebSocket upgrade requests to the same upstream.
</Note>

## Health check

The Superego API exposes a health check at its root path:

```bash
curl https://your-api/
```

```json
{ "name": "superego", "version": "*" }
```

Use this endpoint in your load balancer or container orchestration health checks.

## Environment variables

Copy `.env.example` to `.env` and update the values below for your production environment. Variables marked as required must be set; optional ones have sensible defaults.

### Core settings

| Variable | Required | Description |
|---|---|---|
| `MIKOTO_ENV` | Yes | Set to `PRODUCTION` to enable strict CORS and HSTS headers. Use `DEV` for local development. |
| `SERVER_PORT` | No | Port the Superego API listens on. Defaults to `3511`. |
| `WEB_URL` | Yes | The public URL of the web client (e.g., `https://app.example.com`). Used for CORS and CSP policy in production mode. |

### Database and cache

| Variable | Required | Description |
|---|---|---|
| `DATABASE_URL` | Yes | PostgreSQL connection string: `postgresql://user:pass@host:port/dbname` |
| `AUTOMIGRATE` | No | Set to `true` to run database migrations automatically on startup. |
| `REDIS_URL` | Yes | Redis connection string: `redis://host:port` |

### Authentication

| Variable | Required | Description |
|---|---|---|
| `SECRET` | Yes | Secret key used to sign JWTs. Use a long, randomly generated string in production. |
| `ISSUER` | Yes | JWT issuer claim (e.g., your domain name). |

<Warning>
  Set `SECRET` to a cryptographically random value in production. Anyone who knows this value can forge valid authentication tokens for any user.
</Warning>

### Object storage (S3 / MinIO)

| Variable | Required | Description |
|---|---|---|
| `S3_ACCESS_KEY` | Yes | S3 access key ID. |
| `S3_SECRET_KEY` | Yes | S3 secret access key. |
| `S3_ENDPOINT` | Yes | Hostname of the S3-compatible service (without protocol). |
| `S3_PORT` | Yes | Port of the S3 service. |
| `S3_BUCKET` | Yes | Name of the bucket to store uploaded files in. |
| `S3_USE_SSL` | Yes | Set to `true` when connecting to S3 over HTTPS. |

### Mail

| Variable | Required | Description |
|---|---|---|
| `MAIL_HOST` | Yes | SMTP server hostname. |
| `MAIL_PORT` | Yes | SMTP port. |
| `MAIL_SECURE` | No | Set to `true` to use TLS. |
| `MAIL_USER` | Yes | SMTP username. |
| `MAIL_PASS` | Yes | SMTP password. |
| `MAIL_FROM` | Yes | The `From` address on outgoing emails (e.g., `system@example.com`). |

### LiveKit (voice/video — optional)

| Variable | Required | Description |
|---|---|---|
| `LIVEKIT_SERVER` | No | WebSocket URL of the LiveKit server (e.g., `wss://livekit.example.com`). |
| `LIVEKIT_KEY` | No | LiveKit API key. |
| `LIVEKIT_SECRET` | No | LiveKit API secret. |

### Captcha (optional)

| Variable | Required | Description |
|---|---|---|
| `CAPTCHA_PROVIDER` | No | Set to `disabled` to turn off captcha on registration. |
| `CAPTCHA_SECRET` | No | Server-side captcha secret for your chosen provider. |

### Client / media server

| Variable | Required | Description |
|---|---|---|
| `PUBLIC_MEDIASERVER_URL` | Yes | Public URL of the media server, used by the client to construct upload URLs. |
| `UPLOAD_SECRET` | No | Shared secret for authenticated uploads in production. |
| `CORS_ORIGIN` | No | Restrict media server CORS to this origin in production. |
| `HANDLE_DOMAIN` | Yes | Default domain used for space handles (e.g., `spaces.example.com`). |

## Reverse proxy setup

Route traffic to the application services using your preferred reverse proxy (nginx, Caddy, Traefik, etc.). A typical setup maps subdomains or path prefixes to the appropriate ports:

```
https://app.example.com      → port 3510 (client)
https://api.example.com      → port 3511 (Superego — REST + WebSocket)
https://collab.example.com   → port 3512 (collab)
https://media.example.com    → port 3513 (media server)
```

<Tip>
  The Superego API at port 3511 requires WebSocket support. Ensure your proxy forwards the `Upgrade` and `Connection` headers and does not buffer WebSocket frames. In nginx, include `proxy_http_version 1.1`, `proxy_set_header Upgrade $http_upgrade`, and `proxy_set_header Connection "upgrade"` for the `/ws` location.
</Tip>

When `MIKOTO_ENV=PRODUCTION`, Superego automatically:

- Sets `Strict-Transport-Security` headers (HSTS, 2-year max-age with subdomains).
- Restricts CORS to the `WEB_URL` origin only.
- Applies `Content-Security-Policy`, `X-Frame-Options`, and other security headers.
